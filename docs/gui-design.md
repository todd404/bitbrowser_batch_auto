# GUI 产品重设计

## 一句话结论

现在的 UI 已经能用，但它仍然像“把底层系统做成了页面”。普通用户要先理解 `flow`、`task`、`browser_id`、批次、调度器和运行产物，才能顺畅操作。

下一版要换成产品视角：

```text
我要做什么 -> 要对哪些账号做 -> 要填什么内容 -> 什么时候跑 -> 跑得怎么样 -> 失败怎么处理
```

技术对象继续存在，但只留在高级诊断里。普通用户看到的是：

- 任务模板，而不是 flow 文件。
- 账号窗口，而不是 browser_id。
- 一次运行，而不是底层 task。
- 运行记录和失败处理，而不是 SQLite 状态。
- 截图、输出、错误建议，而不是 JSON 和 trace。

## 产品定位

这个 GUI 不是开发后台，也不是 flow 编辑器。它应该是一个本机批量自动化工作台，帮助用户把已经维护好的比特浏览器账号窗口，按同一套自动化动作批量执行、定时执行、复盘结果。

产品主线只有一条：

```text
创建一次任务 -> 执行 -> 查看结果 -> 处理失败
```

所有页面都应该服务这条主线。任何不能帮助用户完成这条主线的概念，都默认收进“高级”。

## 目标用户与高频场景

### 主要用户

- 已经在比特浏览器里维护了一批账号窗口。
- 经常需要让多个账号执行同一个动作。
- 能理解“账号、窗口、任务、截图、失败重跑”，但不想理解 YAML、Python、调度器和数据库。
- 需要稳定、可复盘，而不是每次都手动盯着命令行。

### 次要用户

- 会写 YAML/Python flow 的高级用户。
- 需要排障、查看 trace、导入任务文件、接入 CLI 的开发者。

### 高频场景

- 临时批量运行：选一批账号，马上执行一个任务。
- 重复定时运行：每天或每周让一批账号自动执行。
- 失败复盘：看哪些账号失败、为什么失败、是否能重跑。
- 账号检查：运行前确认账号窗口是否可用。
- 模板确认：查看某个任务模板需要填哪些内容。

## 当前问题诊断

### 1. 用户必须理解系统设计

当前页面名和字段虽然已经中文化，但用户仍然需要知道：

- “流程”和 YAML/Python 的关系。
- “窗口”其实是账号环境。
- “批次”和“任务”的区别。
- 为什么运行结果里先看到底层任务，再点进去看截图。
- 哪些错误该重跑，哪些错误该先修配置。

这会让普通用户在操作时一直停下来猜：“我现在应该点哪个？”

### 2. 操作路径没有形成心流

创建运行时，用户脑子里通常是：

```text
我要用这些账号做这个动作。
```

但当前向导更像：

```text
选择 flow -> 填 inputs -> 选择 browser -> 配 scheduler。
```

这条路径对开发者合理，对普通用户偏硬。用户需要不断把产品语言翻译成工程语言。

### 3. 错误只展示事实，不给下一步

例如 Local Server 未连接、窗口忙、flow 校验失败、参数缺失、任务失败。当前 UI 更多是在展示状态，下一版要变成：

```text
发生了什么 -> 影响什么 -> 建议点哪里处理
```

### 4. 高级信息出现得太早

JSON、trace、源文件、ID、任务文件路径对排障有用，但不应该占据普通用户的首屏注意力。

## 设计目标

第一版产品化 UI 的目标：

- 用户第一次打开软件，30 秒内知道下一步要做什么。
- 用户不复制任何 ID，也能选择多个账号窗口运行任务。
- 用户不理解 YAML/Python，也能选择“任务模板”并填写参数。
- 用户运行后看到的是进度和结果，不是底层任务表。
- 用户失败后能一键重跑失败项，或看到明确修复建议。
- 高级用户仍能查看源文件、JSON、trace、CLI 等价操作。

## 体验指标

这些指标用来判断产品化改造是否真的有效：

- 首次打开后，用户能在 30 秒内找到“创建任务”入口。
- 创建一次立即运行，普通用户不需要接触任何 ID、JSON 或源文件路径。
- 常见任务从进入创建页到开始运行，不超过 5 个主步骤。
- 创建页每一步最多只有一个主要按钮。
- 失败结果页必须在首屏显示失败数量、失败原因摘要和重跑入口。
- 高级信息默认不出现在首页、创建页和结果页首屏。

## 产品原则

### 1. 一屏只回答一个问题

创建任务时，每一步只问一个核心问题：

```text
做什么？
对谁做？
填什么？
什么时候做？
确认开始？
```

不要在同一步里同时暴露流程类型、窗口状态、参数模式、计划策略和高级 JSON。

### 2. 默认路径要短

最常见路径应该是：

```text
首页 -> 新建任务 -> 选模板 -> 选账号 -> 填参数 -> 开始运行
```

其中高级选项全部有默认值：

- 并发数使用配置默认值。
- 失败重试使用配置默认值。
- 忙碌窗口默认跳过或等待，界面给出提示。
- 运行后是否关闭窗口使用配置默认值。

### 3. 术语站在用户这边

建议术语映射：

| 技术术语 | 普通用户术语 |
| --- | --- |
| flow | 任务模板 |
| browser/window | 账号窗口 |
| browser_id | 窗口 ID，默认隐藏 |
| task | 单个账号的执行项 |
| batch | 一次运行 |
| scheduler | 执行队列 |
| schedule | 定时任务 |
| artifact | 运行文件 |
| trace | 诊断日志 |

页面和主要按钮优先使用普通用户术语。

### 4. 先给结论，再给细节

列表和详情页顶部必须先给结论：

- 当前是否可运行。
- 已选多少账号。
- 预计会创建多少执行项。
- 成功多少、失败多少、还在跑多少。
- 失败是否可以直接重跑。

细节放到下方或折叠区。

### 5. 高级能力渐进展示

高级能力不是删除，而是收纳：

- 源文件路径、flow_type、JSON、trace、browser_id 放到“高级”折叠区。
- 手动导入任务文件放到“设置 > 高级维护”。
- 诊断页面面向排障，不作为普通主路径入口。

## 信息架构

推荐导航：

```text
工作台
创建任务
运行中
运行记录
定时任务
账号窗口
任务模板
设置
```

如果希望导航更短，可以合并为：

```text
工作台
创建任务
运行记录
定时任务
账号窗口
任务模板
设置
```

其中“运行中”可以作为工作台和运行记录里的筛选视图，不一定独立成页面。

### 页面职责

| 页面 | 用户问题 | 页面职责 |
| --- | --- | --- |
| 工作台 | 我现在该做什么？ | 展示就绪状态、正在运行、待处理失败、下一个定时任务、主行动按钮 |
| 创建任务 | 我要批量跑一个任务 | 用向导创建立即运行或定时任务 |
| 运行记录 | 上次跑得怎么样？ | 按“一次运行”查看结果、截图、错误和重跑入口 |
| 定时任务 | 哪些任务会自动跑？ | 查看、启停、立即运行、编辑计划 |
| 账号窗口 | 哪些账号可用？ | 搜索、分组、状态、打开/关闭、健康检测 |
| 任务模板 | 可以做哪些事？ | 查看模板说明、参数、试跑、校验状态 |
| 设置 | 系统怎么配置？ | 普通运行设置和高级诊断 |

## 核心用户心流

### A. 首次打开

首屏不应该是空表格。应该是“工作台”，包含一个轻量就绪区。

展示方式：

```text
比特服务：已连接 / 未连接
账号窗口：找到 12 个 / 未读取到
任务模板：找到 5 个 / 暂无模板
待处理失败：3 个 / 无
```

如果全部正常，只显示一行状态，不占太多空间：

```text
已就绪：12 个账号窗口，5 个任务模板。
```

如果有问题，直接给操作：

- 比特服务未连接：提示“请先启动比特浏览器 Local Server”，按钮“重新检查”。
- 没有账号窗口：按钮“去账号窗口页刷新”。
- 没有任务模板：按钮“查看模板目录”或“打开任务模板页”。
- 有上次中断任务：按钮“恢复运行”或“标记为失败”。

### B. 创建一次立即运行

入口按钮：

```text
创建任务
```

向导步骤：

```text
1. 选择任务模板
2. 选择账号窗口
3. 填写任务内容
4. 运行设置
5. 确认并开始
```

#### 1. 选择任务模板

模板卡片显示：

- 用户可理解的名称：例如“打开网址并截图”。
- 一句话说明：例如“在选中的账号窗口里打开一个网址，并保存最终截图。”
- 标签：网页访问、数据采集、登录检查等。
- 输入摘要：需要填写“目标网址”。
- 最近使用时间或推荐标记。

不在卡片主区域显示：

- declarative / python。
- YAML 文件名。
- 源文件路径。

这些放到模板详情的高级区。

#### 2. 选择账号窗口

选择器要像账号选择器，而不是 ID 表格。

主列建议：

- 序号。
- 窗口名称。
- 分组。
- 备注。
- 状态：可运行 / 正在运行 / 已打开 / 未知。

交互：

- 搜索名称、序号、备注。
- 分组筛选。
- 只看可运行。
- 全选当前筛选结果。
- 保存为常用选择。
- 已选区固定显示“已选 N 个账号”。

窗口 ID 只在详情或复制按钮里出现。

#### 3. 填写任务内容

根据模板元数据生成表单。

默认模式：

```text
所有账号使用同一组内容
```

可切换到：

```text
每个账号使用不同内容
```

每账号不同内容不应该要求用户直接写 JSON。建议分三种方式：

- 表格填写：每行一个账号，每列一个参数。
- 粘贴表格：从 Excel/CSV 复制粘贴。
- 高级 JSON：只放在折叠区。

字段要有用户文案：

```yaml
inputs:
  url:
    type: string
    label: "目标网址"
    required: true
    default: "https://example.com"
    placeholder: "https://example.com"
    help: "每个账号窗口都会打开这个网址。"
```

#### 4. 运行设置

普通设置只展示高频项：

- 立即运行 / 保存为定时任务。
- 并发账号数。
- 失败重试次数。
- 跑完是否关闭窗口。

定时任务再展开时间设置：

- 运行一次。
- 每天。
- 每周。
- 每隔 N 分钟/小时。
- 手动计划。

高级策略放到折叠区：

- 上一次没跑完时怎么处理。
- 错过运行时间怎么处理。
- 忙碌窗口是等待、跳过还是报错。

#### 5. 确认并开始

确认页不要只是“检查通过”。它要用自然语言总结：

```text
将使用“打开网址并截图”，在 12 个账号窗口中运行。
并发 3 个账号，失败最多重试 1 次。
预计会创建 12 个执行项。
```

如果有问题，按严重程度展示：

- 必须处理：没有选择账号、必填参数为空、模板不可用。
- 可以继续：2 个账号正在忙，提交后会等待。
- 建议处理：Local Server 响应较慢。

每个问题都要有按钮或明确下一步：

- “返回选择账号”
- “填写目标网址”
- “重新检查”
- “继续并等待空闲”

### C. 运行中监控

运行中页面按“一次运行”组织。

顶部摘要：

```text
打开网址并截图
12 个账号 | 成功 8 | 失败 1 | 运行中 2 | 等待 1
```

主要操作：

- 暂停新任务。
- 停止等待项。
- 查看失败。
- 跳到结果。

明细表按账号展示：

- 账号窗口。
- 当前状态。
- 当前步骤。
- 已耗时。
- 最后截图。
- 错误摘要。

状态文案要让用户知道系统正在做什么：

| 状态 | 建议文案 |
| --- | --- |
| pending | 等待执行 |
| opening | 正在打开窗口 |
| running | 正在执行任务 |
| success | 已完成 |
| failed | 执行失败 |
| cancelled | 已停止 |

### D. 结果复盘

运行记录默认按“一次运行”展示，而不是按底层 task/run 展示。

列表字段：

- 名称。
- 任务模板。
- 账号数。
- 成功率。
- 状态。
- 创建时间。
- 最近错误。

详情页结构：

```text
结果摘要
失败原因分组
账号明细
截图和输出
高级诊断
```

失败原因分组很重要。用户不应该在 50 行错误里自己找规律。建议聚合：

```text
3 个账号：页面加载超时
2 个账号：目标元素不存在
1 个账号：窗口打开失败
```

每组给操作：

- 重跑这些账号。
- 查看第一个错误。
- 复制错误。

单账号详情优先展示：

- 最后截图。
- 输出字段。
- 错误摘要。
- 重跑这个账号。

高级区再展示：

- run.json。
- trace.json。
- artifact 目录。
- task id。

### E. 定时任务

定时任务不要让用户先理解 cron 或 scheduler。

创建时用自然选择：

```text
什么时候运行？
- 只运行一次
- 每天
- 每周
- 每隔一段时间
- 暂不自动运行，只保存
```

列表字段：

- 任务名称。
- 模板。
- 账号范围。
- 下次运行。
- 最近结果。
- 开关。
- 立即运行按钮。

状态文案：

- 已启用，下一次 2026-07-04 09:30。
- 已停用。
- 只手动运行。
- 上次失败，3 个账号失败。

## 关键页面重设计

### 工作台

布局建议：

```text
顶部：就绪状态 + 创建任务按钮
左侧：正在运行 / 最近运行
右侧：待处理失败 / 下一个定时任务
底部：最近错误或诊断提示
```

工作台不要放太多统计。它的核心任务是把用户带到下一步。

首屏优先级：

1. 如果系统未就绪，显示修复入口。
2. 如果有正在运行，显示进度。
3. 如果有失败，显示重跑入口。
4. 如果都没有，显示“创建任务”。

### 创建任务

这是产品最重要的页面。目标不是“功能全”，而是“用户不迷路”。

改造重点：

- 顶部显示步骤条。
- 下一步按钮固定在底部。
- 每一步右侧显示当前选择摘要。
- 只有当前步骤可编辑，其它步骤折叠成摘要。
- 检查结果只在确认页集中展示。

### 运行记录

运行记录要支持两种视角：

- 最近运行：按时间倒序。
- 待处理失败：只看需要用户动作的记录。

用户常见动作：

- 重跑失败。
- 查看失败截图。
- 导出结果。
- 打开产物目录。

### 账号窗口

窗口页要服务两个目标：

- 运行前选择账号。
- 出问题时检查账号状态。

建议增加：

- 分组筛选。
- 状态筛选。
- 常用选择。
- 批量打开/关闭。
- 健康检测。

### 任务模板

任务模板页不是文件浏览器。它应该告诉用户“能做什么”。

卡片字段：

- 名称。
- 说明。
- 适用场景。
- 需要填写的内容。
- 最近是否校验通过。
- 试跑按钮。

高级区：

- 类型：声明式 / Python。
- 源文件。
- 校验详情。

## Flow 元数据要求

没有好的元数据，UI 就只能显示工程术语。下一版应该把任务模板元数据作为必填体验资产。

声明式 flow 建议：

```yaml
name: "open_and_check"
display_name: "打开网址并截图"
description: "在选中的账号窗口中打开目标网址，并保存最终截图。"
category: "网页访问"
version: 1
inputs:
  url:
    type: string
    label: "目标网址"
    required: true
    default: "https://example.com"
    placeholder: "https://example.com"
    help: "每个账号窗口都会打开这个网址。"
    per_window: false
outputs:
  final_screenshot:
    label: "最终截图"
```

Python flow 建议使用同名 `.meta.yaml`：

```text
flows/py/paginate_titles.py
flows/py/paginate_titles.meta.yaml
```

不要为了读取 UI 信息 import Python flow。元数据应是静态文件。

## 文案规范

### 按钮文案

建议：

- 创建任务
- 开始运行
- 保存为定时任务
- 重跑失败
- 查看结果
- 重新检查
- 打开窗口
- 关闭窗口

避免：

- 提交
- 执行调度器
- 导入 tasks
- 查看 artifact
- validate flow

### 错误文案

错误要包含三段：

```text
发生了什么。
这会影响什么。
现在可以怎么做。
```

示例：

```text
未连接到比特浏览器 Local Server。
当前无法读取账号窗口，也无法开始运行。
请先启动比特浏览器本地服务，然后点击“重新检查”。
```

### 空状态文案

空状态要给下一步。

```text
还没有运行记录。
创建第一个任务后，这里会显示每次运行的结果、截图和错误。
```

## 与当前代码的映射

当前实现已经有可复用底座：

- `UiCoreService.list_flow_cards()` 可以支撑任务模板卡片。
- `UiCoreService.list_browser_windows()` 可以支撑账号窗口选择器。
- `UiCoreService.create_batch_run()` 可以支撑一次运行。
- `Storage.batch_runs` 可以支撑运行记录。
- `Storage.schedules` 可以支撑定时任务。

产品化改造不需要推翻核心，只需要重排 UI 层：

```text
core engine: BitBrowserClient / Scheduler / Runner / Storage
product service: UiCoreService 面向用户任务提供方法
ui layer: NiceGUI 页面与组件
advanced layer: JSON、trace、任务文件、CLI 等价命令
```

## 推荐 UI 组件拆分

当前 `src/bitbrowser_auto/ui/app.py` 已经偏大，继续堆逻辑会让产品迭代变慢。建议拆分：

```text
bitbrowser_auto/ui/
  app.py
  core.py
  state.py
  components/
    layout.py
    readiness.py
    task_template_card.py
    account_picker.py
    input_form.py
    run_summary.py
    result_detail.py
  pages/
    home.py
    create_task.py
    running.py
    records.py
    schedules.py
    accounts.py
    templates.py
    settings.py
```

`app.py` 只负责启动、主题和路由。页面之间通过 `UiCoreService` 交互。

## 分阶段落地

### Phase UX-1: 术语和首屏改造

目标：用户打开后知道下一步。

交付：

- 导航改成“工作台 / 创建任务 / 运行记录 / 定时任务 / 账号窗口 / 任务模板 / 设置”。
- 首页按就绪状态、正在运行、待处理失败、主行动按钮重排。
- 普通页面隐藏 ID、JSON、trace。
- 设置拆成“运行设置”和“高级诊断”。

验收：

- 用户不理解 `flow`、`task`、`browser_id` 也能知道从“创建任务”开始。
- Local Server 异常时不显示堆栈，显示可执行建议。

### Phase UX-2: 创建任务向导 v2

目标：创建任务形成自然心流。

交付：

- 任务模板卡片。
- 账号窗口选择器。
- 表单参数区。
- 每账号不同参数的表格模式。
- 确认页自然语言总结和可处理检查项。

验收：

- 用户能在 5 步内创建一次立即运行。
- 用户不需要复制窗口 ID。
- 用户不需要写 JSON 就能完成常见批量任务。

### Phase UX-3: 运行中和结果复盘

目标：运行后用户能看懂状态并处理失败。

交付：

- 运行中批次页。
- 失败原因分组。
- 截图优先的单账号详情。
- 重跑失败、重跑单账号、导出结果。

验收：

- 用户能一眼看到成功率和失败数。
- 用户能一键重跑失败账号。
- 用户能打开截图和错误，不需要先看 trace。

### Phase UX-4: 定时任务产品化

目标：定时任务不用理解 scheduler。

交付：

- 自然语言定时创建。
- 启停开关。
- 立即运行。
- 最近结果和下次运行展示。
- 重叠策略、错过策略高级化。

验收：

- 用户能创建一次性、每天、每周、间隔运行计划。
- 用户能看懂下次什么时候跑。
- UI 重启后计划仍存在。

### Phase UX-5: 桌面体验

目标：普通用户双击可用。

交付：

- Desktop Mode 默认入口。
- 关闭窗口时检测运行中任务并确认。
- 首次启动检查。
- 打包后默认配置。

验收：

- 干净机器上能双击启动。
- 未连接 Local Server 时提示清晰。
- 关闭 UI 不会静默中断正在运行的任务。

## 第一版验收清单

- 打开软件后，用户能看到系统是否就绪。
- 用户能从首页进入“创建任务”。
- 用户能选择一个任务模板。
- 用户能按名称、序号、备注选择多个账号窗口。
- 用户能填写所有账号共用的参数。
- 用户能用表格方式填写每账号不同参数。
- 用户能立即运行。
- 用户能保存定时任务。
- 用户能查看正在运行的一次运行。
- 用户能按一次运行查看历史记录。
- 用户能看到失败原因和截图。
- 用户能一键重跑失败账号。
- 高级用户能在折叠区查看源文件、JSON、trace 和底层任务。

## 非目标

本轮不做：

- 可视化 flow 编排器。
- 内置 flow 编写 Agent。
- 多用户权限系统。
- 云端队列。
- flow 市场。
- 未信任 Python 代码沙箱。

这些以后可以做，但当前最重要的是让本机普通用户顺畅完成批量运行、定时运行和失败复盘。
